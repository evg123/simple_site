Post Mortem

Adding multiplayer to my platformer was quite challenging, and while I was ultimately successful, the final product is not as polished as I would have liked. Starting with my implementation of a platformer from earlier in the semester and adapting it to fit a client-server model shed some light on how my original design could be improved. If the different subsystems had been properly decoupled it would have been much simpler to split the client and server into separate code paths.

I think in future projects I will be sure to keep things like physics updates and object death/removal completely separate from the code that manages the overall state of the game. Using a component system also would have been much cleaner, as I could have simply not added the graphics, audio, etc components to game objects when running in dedicated server mode.

That said, I did get some valuable practical experience with network programming for games. I implemented client-side prediction and allowed a client to notice and adjust based on discrepancies between updates from the server and the client's state at the timestamp that the update represents. I also implemented a network clock synchronization algorithm that samples the round-trip-time of communication with the server to estimate the offset between the client and server clocks.
